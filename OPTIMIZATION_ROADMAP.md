# 优化路线图

## 概述

本文档记录智能竞品分析 Agent 的优化方向和实施计划。

## 优先级说明

- **P0 (高优先级)**: 投入产出比高，应尽快实施
- **P1 (中优先级)**: 重要但不紧急，可规划实施
- **P2 (低优先级)**: 长期规划，需要前置条件

---

## 1. 架构优化 [P0/P1]

### 1.1 流程并行化 [P1]

**现状**: 6个节点串行执行，多竞品分析耗时长

**优化方案**:
- 多竞品并行搜索/爬取
- 使用 `asyncio.gather` 并发执行独立任务
- 预计可节省 50-70% 的采集时间

**实现要点**:
```python
# 示例：并行爬取多个竞品
async def scrape_data(state: AgentState):
    tasks = [scrape_competitor(comp) for comp in state["competitors"]]
    results = await asyncio.gather(*tasks)
    return {"raw_data": results}
```

### 1.2 条件分支 [P1]

**现状**: 固定线性流程，所有分析类型执行相同步骤

**优化方案**:
- quick 分析跳过爬取，直接用搜索摘要
- deep 分析增加更多数据源
- 根据数据质量动态决定是否重试

**实现要点**:
```python
graph.add_conditional_edges(
    "planner",
    lambda state: state["analysis_type"],
    {
        "quick": "extractor",   # 跳过搜索和爬取
        "standard": "searcher",
        "deep": "searcher"
    }
)
```

### 1.3 重试机制 [P0]

**现状**: 失败直接降级，无重试

**优化方案**:
- 节点级重试（最多3次）
- 指数退避策略
- 区分可重试错误和不可重试错误

### 1.4 缓存层 [P0]

**现状**: 无缓存，重复分析相同竞品时重复请求

**优化方案**:
- 搜索结果缓存（TTL: 24小时）
- 网页内容缓存（TTL: 7天）
- 使用 Redis 或本地文件缓存

**收益**:
- 减少 SerpAPI 调用次数（节省成本）
- 加快重复分析速度
- 支持离线分析

## 3. LLM 优化 [P0/P1]

### 3.1 Prompt 工程优化 [P0]

**现状**: Prompt 模板较为基础

**优化方案**:
- 添加 Few-shot 示例
- 优化输出格式约束
- 针对不同行业定制 Prompt
- A/B 测试评估 Prompt 效果

**涉及文件**:
- `config/prompts.py` - 所有 Prompt 模板

### 3.2 模型选择策略 [P1]

**现状**: 所有节点使用相同模型

**优化方案**:
| 节点 | 推荐模型 | 原因 |
|------|----------|------|
| planner | 快速模型 | 规划任务简单 |
| extractor | 快速模型 | 提取任务结构化 |
| analyzer | 强模型 | 需要推理能力 |
| reporter | 强模型 | 需要写作能力 |

### 3.3 结构化输出 [P1]

**现状**: LLM 输出 JSON 后手动解析

**优化方案**:
- 使用 Pydantic 定义输出 Schema
- 使用 Instructor 或 LMQL 约束输出
- 减少 JSON 解析失败率

### 3.4 流式响应 [P1]

**现状**: 等待完整报告生成后返回

**优化方案**:
- 报告生成支持流式输出
- 前端实时显示生成进度
- 改善用户体验

### 3.5 成本控制 [P1]

**优化方案**:
- Token 使用统计
- 预算限制和告警
- 缓存减少重复调用

## 7. 部署与运维 [P1]

### 7.1 容器化 [P1]

**方案**:
```dockerfile
# Dockerfile 示例
FROM python:3.13-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
CMD ["uvicorn", "api.app:app", "--host", "0.0.0.0", "--port", "8000"]
```

**docker-compose.yml**:
- api 服务
- frontend 服务
- redis 服务（缓存）
- chroma 服务（向量数据库）

### 7.2 任务队列 [P1]

**方案**: Celery + Redis

**收益**:
- 异步处理长时任务
- 任务重试和失败处理
- 任务优先级和限流

### 7.3 监控告警 [P1]

**监控指标**:
- API 响应时间
- 分析任务成功率
- LLM 调用延迟和成本
- 错误率和异常

**告警规则**:
- 分析失败率 > 10%
- API 响应时间 > 5s
- 任务队列积压 > 100

### 7.4 API 限流 [P1]

**方案**:
- 基于 IP 的限流
- 基于用户的配额
- 突发流量控制
